1¿Qué es MagicMirror² y por qué lo elegimos?

MagicMirror² es un framework de código abierto (licencia MIT) que convierte cualquier pantalla en un espejo inteligente. Funciona como una aplicación Node.js + Electron: Node.js gestiona el servidor web y la lógica de negocio, mientras Electron renderiza la interfaz en Chromium a pantalla completa.

¿Por qué MagicMirror² y no una solución personalizada?

Porque ya resuelve los problemas difíciles: comunicación en tiempo real entre módulos (Socket.io), carga dinámica de módulos, acceso remoto por red y un ecosistema de más de 1.500 módulos de la comunidad. Construir todo eso desde cero habría sido inviable en el tiempo disponible.

🖥️

Un monitor detrás de un espejo semitransparente. El fondo negro del espejo no refleja nada; los elementos blancos/luminosos sí son visibles.

🧩

2Arquitectura técnica del sistema

El sistema tiene tres capas bien diferenciadas que se comunican entre sí:

CAPA 1 — DATOS (fuentes externas)
  OpenWeatherMap API ── tiempo actual + previsión 5 días (REST/JSON)
  Spotify API ── reproducción en curso (OAuth 2.0 + polling)
  Yahoo Finance ── cotizaciones BTC, DOGE, Petróleo (scraping interno)
  RSS Feeds ── El País · La Vanguardia · RTVE (XML/RSS)

CAPA 2 — SERVIDOR (Node.js · puerto 8080)
  node_helper.js de cada módulo → hace las peticiones HTTP a las APIs externas
  Express → sirve archivos estáticos (HTML, CSS, JS, imágenes)
  Socket.io server → envía los datos al navegador en tiempo real
  API REST (MMM-Remote-Control) → recibe comandos del exterior (cambio de página, etc.)

CAPA 3 — INTERFAZ (Electron / Chromium)
  Socket.io client → recibe datos del servidor y los pinta en DOM
  MMM-pages → gestiona visibilidad CSS de módulos por página
  Módulos JS → generan el HTML de cada widget (reloj, clima, etc.)

CONTROL REMOTO (cualquier dispositivo de la red)
  PAGINAS.html → fetch() → POST /api/notification/PAGE_CHANGED
  curl / script → POST /api/notification/... (automatización)

4Problemas reales encontrados y cómo se resolvieron

Estos no son problemas teóricos: ocurrieron durante el desarrollo y requirieron investigación y decisiones técnicas concretas.

Problema 1MMM-SmartWebDisplay no mostraba las imágenes. El iframe cargaba en blanco.
Solución Los CDNs externos usan la cabecera HTTP X-Frame-Options: DENY para impedir ser incrustados en iframes por seguridad. Se creó img.html, un archivo HTML mínimo servido por el propio MagicMirror en localhost que carga la imagen con una etiqueta <img> en lugar de un iframe. Al estar en el mismo origen (localhost:8080), no hay restricciones de seguridad.
Problema 2El parámetro de configuración del módulo se llamaba urls (plural) pero el módulo no funcionaba.
Solución Leyendo el código fuente de MMM-SmartWebDisplay se descubrió que el módulo internamente lee this.config.url (singular). La documentación estaba desactualizada. Además, los elementos del array debían ser strings planos, no objetos { url: "...", label: "..." }. Se corrigió en config.js y funciona.
Problema 3MMM-OpenWeatherForecast se quedaba en "Cargando..." sin mostrar datos.
Solución Desde 2022, OpenWeather requiere registrar una tarjeta de crédito para activar su API One Call 3.0, incluso en el plan gratuito. Se decidió sustituir el módulo por el módulo nativo weather de MagicMirror², que usa los endpoints data/2.5/weather y data/2.5/forecast, que son completamente gratuitos y no requieren tarjeta. La funcionalidad es equivalente.
Problema 4PAGINAS.html abría desde VS Code Live Server (puerto 5500) pero no podía controlar el espejo.
Solución Entender que son dos servidores distintos: Live Server sirve archivos estáticos desde el disco en :5500; MagicMirror tiene su propio servidor Express en :8080. PAGINAS.html funciona desde cualquier servidor porque sus botones hacen peticiones fetch() directamente a http://192.168.18.112:8080/api/.... El HTML puede estar en cualquier sitio siempre que MagicMirror esté corriendo en :8080.
Problema 5MMM-Spotify aparecía en blanco sin ningún mensaje de error.
Solución El módulo requiere autenticación OAuth 2.0 con Spotify antes del primer uso. Se creó auth_windows.js, un script que lanza un servidor HTTP local en el puerto 8888, genera la URL de autorización de Spotify, espera el callback con el código de autorización, lo intercambia por un token y lo guarda en un archivo JSON. A partir de ahí el módulo lo usa y lo renueva automáticamente.

5Decisiones de diseño y por qué

  • Sistema de 3 páginas en lugar de un dashboard único
    Con todos los módulos visibles a la vez la pantalla se satura. El sistema de páginas (MMM-pages) permite mostrar información relevante según el momento: inicio del día (reloj + clima), durante el ocio (música + finanzas), o modo galería. La navegación es inmediata y sin recargar.
  • Módulos fijos para información siempre crítica
    Las noticias, el indicador de página y el control remoto permanecen visibles en todas las páginas. Son los módulos de "infraestructura": control de navegación y contexto informativo continuo independientemente de la página activa.
  • API REST abierta en red local (sin autenticación)
    Se configuró secureEndpoints: false en MMM-Remote-Control porque el espejo opera en una red local privada. No tiene sentido añadir autenticación que complique el uso desde el móvil o la tablet. En producción pública sería diferente.
  • Carpeta imagenes_a_mostrar/ para la galería
    Separar las imágenes del código del módulo facilita la gestión: añadir, eliminar o sustituir fotos sin tocar ningún archivo de código. Solo hay que copiar o renombrar archivos en esa carpeta y actualizar las URLs en config.js.
  • data/2.5 en lugar de data/3.0 para el tiempo
    La API 3.0 requiere tarjeta de crédito aunque sea gratuita. La API 2.5 ofrece tiempo actual y previsión de 5 días, que es todo lo que necesita el espejo. La decisión fue pragmática: funcionalidad equivalente, cero fricción para quien instale el proyecto.
  • PM2 para el arranque automático
    PM2 es un gestor de procesos para Node.js que reinicia la aplicación si falla y la lanza automáticamente al arrancar el sistema. Alternativa más ligera que systemd para este caso de uso. Permite ver logs, reiniciar y parar con un solo comando.

6El flujo OAuth 2.0 de Spotify explicado

OAuth 2.0 es un protocolo de autorización que permite a una aplicación acceder a recursos de un usuario sin que el usuario entregue su contraseña a la aplicación. Es el estándar de la industria.

AUTHORIZATION CODE FLOW (el que usa este proyecto)

1. auth_windows.js genera URL → accounts.spotify.com/authorize?client_id=...&scope=...&redirect_uri=...
2. Usuario abre la URL en el navegador y hace clic en "Acepto"
3. Spotify redirige a http://127.0.0.1:8888/callback?code=CODIGO_UNICO
4. auth_windows.js captura el code y hace POST a accounts.spotify.com/api/token
5. Spotify devuelve access_token (válido 1h) + refresh_token (para renovar)
6. Ambos tokens se guardan en MagicMirror_token.json
7. MMM-Spotify usa el access_token y lo renueva automáticamente con el refresh_token

→ El usuario solo interviene en los pasos 2 y 3. Todo lo demás es automático.

8.1 · config.js — el centro de todo

config.js es el único archivo que hay que tocar para configurar el espejo completo. MagicMirror lo lee al arrancar y a partir de él sabe qué módulos cargar, con qué opciones y en qué posición de pantalla.

Estructura del archivo config.js

let config = {
  address: "0.0.0.0" ← IP de escucha. 0.0.0.0 = acepta conexiones de toda la red local
  port: 8080 ← Puerto HTTP del servidor Express de MagicMirror
  ipWhitelist: [] ← [] = no filtrar IPs, cualquier dispositivo puede acceder
  language: "es" ← Idioma de los módulos nativos (fechas, textos)
  locale: "es-EU" ← Formato de números y fechas (europeo: 1.234,56)
  timeFormat: 24 ← Formato de hora. 24 = sin AM/PM
  units: "metric" ← Sistema de unidades: metric = °C, km/h
  logLevel: [...] ← Qué mensajes aparecen en la consola al ejecutar

  modules: [ ← Array de objetos. Cada objeto = un módulo activo
    {
      module: "nombre-del-modulo" ← Nombre exacto de la carpeta en modules/
      position: "top_left" ← Zona de pantalla donde se renderiza
      header: "Mi título" ← Título opcional sobre el módulo
      config: { ... } ← Opciones propias del módulo (cada uno tiene las suyas)

¿Por qué el nombre del módulo es sensible a mayúsculas?

MagicMirror busca la carpeta del módulo en modules/ usando exactamente el nombre escrito en module:. Si escribes "mmm-pages" en lugar de "MMM-pages", en sistemas Linux/macOS el sistema de archivos distingue mayúsculas y no encontrará la carpeta. En Windows no falla, pero en Raspberry Pi (Linux) sí. Por eso hay que respetar el nombre exacto.

8.2 · Anatomía de un módulo — qué archivos tiene y para qué sirve cada uno

Cada módulo es una carpeta dentro de modules/. Su estructura mínima tiene dos archivos clave:

modules/NombreModulo/
  NombreModulo.js ← Código del FRONTEND. Se ejecuta en el navegador (Electron/Chromium)
  node_helper.js ← Código del BACKEND. Se ejecuta en Node.js (servidor)
  NombreModulo.css ← Estilos visuales del módulo (opcional)
  package.json ← Dependencias npm del módulo (axios, moment, etc.)
  node_modules/ ← Dependencias instaladas con npm install (no tocar)
🖥️

NombreModulo.js (frontend)

Se ejecuta dentro de Chromium/Electron. Su función principal es construir el HTML que aparece en pantalla y recibir datos del backend via Socket.io.

⚙️

node_helper.js (backend)

Se ejecuta en Node.js. Su función es hacer peticiones HTTP a APIs externas (clima, Spotify, etc.) y enviar los datos al frontend via Socket.io.

Funciones del archivo de módulo (frontend — NombreModulo.js)

FunciónCuándo se llamaQué hace
start() Al arrancar MagicMirror, una vez Inicialización del módulo. Aquí se inicializan variables internas y se envía la primera petición de datos al node_helper.
getDom() Cada vez que hay que re-renderizar el módulo Devuelve un elemento HTML (div, table, etc.) que MagicMirror pintará en la posición configurada. Es la función más importante del frontend.
getStyles() Al cargar el módulo Devuelve un array con rutas de archivos CSS a cargar para este módulo.
getHeader() Al renderizar Devuelve el texto del título del módulo (lo que aparece encima del widget).
socketNotificationReceived(notification, payload) Cuando el node_helper envía datos Recibe el nombre de la notificación y los datos. Aquí el módulo guarda los datos y llama a this.updateDom() para redibujar.
notificationReceived(notification, payload, sender) Cuando otro módulo envía un evento Permite que módulos se comuniquen entre sí. MMM-pages lo usa para mostrar/ocultar módulos cuando cambia la página.

Funciones del node_helper.js (backend)

FunciónCuándo se llamaQué hace
start() Al arrancar el servidor de MagicMirror Inicialización del helper. Normalmente solo imprime un log de confirmación.
socketNotificationReceived(notification, payload) Cuando el frontend envía una petición El módulo frontend llama a this.sendSocketNotification("PEDIR_DATOS", {...}) y aquí en el backend se recibe ese mensaje. El helper hace la petición HTTP a la API externa y devuelve los datos.

El ciclo completo de datos en un módulo (ejemplo: módulo weather)
1. weather.js llama this.sendSocketNotification("WEATHER_GET", { apiKey, lat, lon })
2. node_helper.js recibe "WEATHER_GET" → hace axios.get("api.openweathermap.org/...")
3. La API de OpenWeather responde con un JSON con temperatura, viento, etc.
4. node_helper.js llama this.sendSocketNotification("WEATHER_DATA", responseJSON)
5. weather.js recibe "WEATHER_DATA" → guarda los datos → llama this.updateDom()
6. MagicMirror llama a getDom() → el módulo construye el HTML con los datos nuevos
7. MagicMirror reemplaza el contenido en pantalla con el nuevo HTML

8.3 · Socket.io — por qué la pantalla se actualiza sola

Socket.io es una librería que crea un canal de comunicación bidireccional y persistente entre el servidor (Node.js) y el cliente (Chromium/Electron). A diferencia de HTTP normal donde el cliente siempre pregunta y el servidor responde, con Socket.io el servidor puede enviar datos al cliente cuando quiera, sin que el cliente lo pida.

HTTP normal (sin tiempo real)
Cliente ──── GET /datos ────→ Servidor
Cliente ←── respuesta ────── Servidor
(el cliente tiene que preguntar cada vez; el servidor no puede iniciar)

Socket.io (tiempo real)
Cliente ══════ WebSocket ══════ Servidor
Cliente ──── emit("pedir") ──→ Servidor
Cliente ←── emit("datos") ─── Servidor ← el servidor envía cuando tiene datos nuevos
Cliente ←── emit("datos") ─── Servidor ← y otra vez... sin que el cliente pregunte

En MagicMirror, cada módulo se comunica con su node_helper a través de Socket.io usando notificaciones con nombre. Por ejemplo, el módulo del tiempo envía "WEATHER_GET" y su helper responde con "WEATHER_DATA". Los nombres son strings arbitrarios — son una convención del propio módulo.

8.4 · MMM-pages — cómo funciona la navegación entre páginas

MMM-pages no mueve módulos ni recarga la página. Su truco es más simple: cambia el CSS de visibilidad (display: none / display: block) de cada módulo según la página activa. Todos los módulos están siempre cargados en memoria — solo se ocultan o muestran visualmente.

config.js — cómo se definen las páginas en MMM-pages

modules: [
  ["clock", "compliments", "weather"] ← Página 1 (índice 0): estos módulos son visibles
  ["MMM-Jast", "MMM-Spotify"] ← Página 2 (índice 1)
  ["MMM-SmartWebDisplay"] ← Página 3 (índice 2)
],

fixed: ["alert", "MMM-Remote-Control", "MMM-Page-Indicator", "newsfeed"]
← Estos módulos tienen display:block SIEMPRE, independientemente de la página

homePage: 0 ← Al arrancar, empieza en la página de índice 0 (Página 1)
timings: { default: 0 } ← 0 = sin rotación automática. Si fuera 30000 = rota cada 30 segundos
animationTime: 1000 ← La transición entre páginas dura 1 segundo (fade)

Cuando se recibe la notificación PAGE_CHANGED con payload 1, MMM-pages oculta todos los módulos de la página actual y muestra los de la página 1 (índice 1 = Finanzas). Los módulos del array fixed nunca se tocan.

⚠ Por qué los nombres en modules[] deben coincidir exactamente

MMM-pages identifica cada módulo por su campo module: en config.js. Si en modules[] escribes "mmm-jast" pero en el módulo está declarado como "MMM-Jast", MMM-pages no lo encontrará y el módulo será visible en todas las páginas a la vez (como si no estuviera en ninguna).

8.5 · API REST de MMM-Remote-Control — cómo controlar el espejo desde fuera

MMM-Remote-Control añade a Express (el servidor de MagicMirror) un conjunto de rutas HTTP adicionales que permiten controlar el espejo desde cualquier dispositivo de la red. Una API REST es simplemente un conjunto de URLs a las que se puede llamar con peticiones HTTP estándar.

Método HTTPRutaQué haceEjemplo de body
POST/api/notification/PAGE_CHANGEDCambia a la página indicada{"payload": 1}
POST/api/notification/PAGE_INCREMENTAvanza una página—
POST/api/notification/PAGE_DECREMENTRetrocede una página—
POST/api/notification/SPOTIFY_PAUSEPausa Spotify—
GET/api/moduleLista todos los módulos cargados—
GET/api/Información general de la API—

Cuando la API recibe un POST /api/notification/PAGE_CHANGED, internamente llama a sendNotification("PAGE_CHANGED", payload) dentro del sistema de MagicMirror, que propaga el evento a todos los módulos — incluyendo MMM-pages, que lo escucha y ejecuta el cambio de página.

Flujo de un cambio de página desde PAGINAS.html

1. Usuario pulsa botón "Finanzas" en PAGINAS.html
2. JavaScript ejecuta: fetch('http://192.168.18.112:8080/api/notification/PAGE_CHANGED', { method:'POST', body:'{"payload":1}' })
3. Express recibe la petición → MMM-Remote-Control la procesa
4. Se emite internamente: sendNotification("PAGE_CHANGED", 1)
5. MMM-pages recibe la notificación → oculta módulos página 0 → muestra módulos página 1
6. La pantalla del espejo cambia en menos de 1 segundo
7. La API responde HTTP 200 → PAGINAS.html muestra el botón como activo

8.7 · Spotify OAuth 2.0 — cada línea de auth_windows.js explicada

OAuth 2.0 con Authorization Code Flow es el método más seguro para que una app acceda a la cuenta Spotify de un usuario. Nunca se transmite la contraseña del usuario a la app.

TérminoQué esEn nuestro proyecto
CLIENT_ID Identificador público de la aplicación registrada en Spotify Developer Guardado en spotify.config.json. No es secreto — Spotify lo usa para saber qué app pide permiso
CLIENT_SECRET Clave secreta de la aplicación. Como una contraseña de la app (no del usuario) También en spotify.config.json. Solo se envía al servidor de Spotify, nunca al navegador
REDIRECT_URI URL a la que Spotify redirige al usuario después de autorizar. Debe estar registrada en el Dashboard http://127.0.0.1:8888/callback — el servidor que lanza auth_windows.js
scope Lista de permisos que la app pide al usuario user-read-playback-state user-modify-playback-state streaming — ver y controlar la reproducción
code Código temporal (válido pocos segundos) que Spotify envía tras la autorización Se recibe en el callback y se intercambia inmediatamente por el token
access_token Token de acceso. Se incluye en cada petición a la API de Spotify como cabecera Authorization: Bearer ... Guardado en el archivo JSON de token. Válido 1 hora
refresh_token Token permanente para obtener nuevos access_tokens sin pedir permiso de nuevo También en el JSON. MMM-Spotify lo usa automáticamente cuando el access_token expira
Lo que hace auth_windows.js paso a paso

1. Lee spotify.config.json → obtiene CLIENT_ID, CLIENT_SECRET y ruta del token
2. Comprueba si ya existe el archivo de token → si existe, sale (ya está autenticado)
3. Construye la URL de autorización:
  https://accounts.spotify.com/authorize
  ?response_type=code ← pedir código de autorización
  &client_id=... ← identificar la app
  &scope=... ← permisos solicitados
  &redirect_uri=http://127.0.0.1:8888/callback
4. Lanza un servidor HTTP en localhost:8888 con http.createServer()
5. Muestra la URL en consola → el usuario la abre y acepta en el navegador
6. Spotify redirige a localhost:8888/callback?code=XXXX
7. El servidor captura el code de los parámetros de la URL
8. Hace POST a accounts.spotify.com/api/token con el code + credenciales
9. Spotify devuelve access_token + refresh_token en JSON
10. Calcula expires_at = ahora + (expires_in - 60) segundos → por seguridad, renueva 1 min antes
11. Guarda el token en MagicMirror_token.json → cierra el servidor → proceso terminado

8.8 · img.html — la solución al bloqueo de iframes

MMM-SmartWebDisplay muestra contenido dentro de un <iframe>. El problema: muchos servidores web (incluidos CDNs de imágenes) envían la cabecera HTTP X-Frame-Options: DENY o Content-Security-Policy: frame-ancestors 'none', que instruye al navegador a negarse a mostrar ese contenido dentro de un iframe. Es una medida de seguridad anti-clickjacking.

¿Qué es clickjacking? (por qué existe X-Frame-Options)
Un atacante incrusta tu banco dentro de un iframe transparente encima de un botón falso.
El usuario cree que hace clic en "Descargar" pero en realidad pulsa "Transferir dinero" en el banco.
X-Frame-Options evita exactamente esto: el banco dice "no me metas en iframes".

La solución: en lugar de intentar cargar la imagen del CDN directamente en el iframe, se crea img.html, un archivo HTML mínimo servido por nuestro propio servidor (localhost:8080). Dentro de ese archivo, una etiqueta <img> carga la imagen.

Contenido completo de img.html

<!DOCTYPE html>
<html><head>
  <style>
    * { margin: 0; padding: 0; } ← sin márgenes
    body { background: #000; display: flex;
          align-items: center; justify-content: center; } ← imagen centrada
    img { max-width: 100%; max-height: 100%; object-fit: contain; }
  </style>
</head><body>
  <img id="pic" src=""> ← imagen vacía al principio
  <script>
    const src = new URLSearchParams(location.search).get('src')
    // URLSearchParams lee los parámetros de la URL (?src=http://...)
    // location.search es la parte "?src=..." de la URL actual
    // .get('src') extrae el valor del parámetro llamado "src"
    if (src) document.getElementById('pic').src = src
    // Si hay valor, lo asigna al atributo src de la etiqueta img
  </script>
</body></html>

La URL completa que usa config.js es:

http://localhost:8080/modules/MMM-SmartWebDisplay/img.html?src=http://localhost:8080/modules/MMM-SmartWebDisplay/imagenes_a_mostrar/img1.jpg

El iframe carga img.html (mismo origen: localhost:8080 → sin restricciones). Ese HTML, con JavaScript, extrae la ruta de la imagen del parámetro src y la carga con <img>. Una imagen con <img> no tiene restricciones de origen cruzado.

8.9 · RSS — cómo se consumen las noticias

RSS (Really Simple Syndication) es un formato estándar basado en XML que los medios de comunicación usan para publicar sus titulares. El módulo newsfeed descarga estos archivos XML y extrae los titulares para mostrarlos en rotación.

Estructura simplificada de un RSS

<rss version="2.0">
  <channel>
    <title>El País</title>
    <item>
      <title>Pedro Sánchez anuncia nuevas medidas...</title>
      <link>https://elpais.com/articulo/1234</link>
      <pubDate>Mon, 17 Mar 2026 10:30:00 GMT</pubDate>
    </item>
    <item> ... </item>
  </channel>
</rss>

El módulo descarga el XML cada reloadInterval milisegundos (300.000 ms = 5 minutos), lo parsea y pone los titulares en cola. Cada updateInterval (7.500 ms = 7,5 segundos) muestra el siguiente titular con una animación de fade.

8.10 · MMM-Jast — cotizaciones sin API key

Yahoo Finance ofrece datos de cotizaciones a través de una URL no oficial que devuelve JSON. MMM-Jast construye esa URL con el símbolo del activo y la consume directamente. Por eso no necesita registro ni API key: Yahoo Finance no requiere autenticación para consultas básicas.

Símbolos configurados y por qué

BTC-EUR ← Bitcoin cotizado en euros. El sufijo "-EUR" indica la divisa
DOGE-EUR ← Dogecoin en euros
CL=F ← Petróleo Crudo WTI (futuros). El sufijo "=F" indica contrato de futuros en Yahoo Finance. Cotiza en USD

El módulo actualiza cada updateIntervalInSeconds: 30 (30 segundos) y muestra los cambios en verde (subida) o rojo (bajada) con el porcentaje de variación diaria.

8.11 · PM2 — por qué se usa y qué hace exactamente

PM2 (Process Manager 2) es un gestor de procesos para Node.js. Su función es mantener la aplicación viva: si MagicMirror falla o el sistema se reinicia, PM2 lo relanza automáticamente.

Problema sin PM2Solución con PM2
Si cierras la terminal, MagicMirror se para PM2 lo ejecuta como demonio (proceso en segundo plano, independiente de la terminal)
Si el sistema se reinicia, hay que lanzar MagicMirror manualmente pm2 startup registra PM2 como servicio del sistema operativo → arranca solo al iniciar el PC
Si hay un error que mata el proceso, el espejo se apaga PM2 detecta que el proceso murió y lo reinicia automáticamente
Difícil ver qué está pasando en segundo plano pm2 logs MagicMirror muestra los logs en tiempo real aunque no haya terminal abierta
Comandos PM2 más usados en este proyecto

pm2 start npm --name "MagicMirror" -- run server
→ Lanza MagicMirror en modo servidor (sin Electron). El nombre "MagicMirror" es el alias.
→ "-- run server" es el argumento que se pasa a npm (equivale a "npm run server")

pm2 save
→ Guarda la lista de procesos activos para que pm2 startup los recuerde

pm2 startup
→ Genera e imprime un comando sudo que registra PM2 en el sistema operativo
→ Hay que copiar y ejecutar ese comando manualmente (requiere permisos de root)

pm2 logs MagicMirror ← Ver qué está pasando en tiempo real
pm2 restart MagicMirror ← Aplicar cambios en config.js sin reiniciar el sistema
pm2 stop MagicMirror ← Parar el espejo sin matar PM2
pm2 status ← Ver estado de todos los procesos gestionados

9Configuración detallada — cada parámetro explicado

Aquí se detalla cada parámetro de cada módulo: qué hace, por qué tiene ese valor y qué pasaría si lo cambias. Es la parte más densa pero la más útil para responder preguntas técnicas.

9.1 · clock — reloj digital

El módulo clock es nativo de MagicMirror². No necesita ninguna instalación adicional ni API key. Se renderiza en top_left para ocupar la esquina superior izquierda, posición tradicional del reloj en un espejo.

ParámetroValor usadoQué haceAlternativas
displayType "digital" Muestra el reloj en formato numérico digital (solo números). La pantalla retroiluminada maximiza la legibilidad con texto grande y claro. "analog" muestra un reloj de agujas. "both" muestra los dos. Se eligió digital porque ocupa menos espacio y es más legible desde distancia.
timeFormat 24 Usa formato de 24 horas (13:45 en lugar de 1:45 PM). Coherente con el estándar europeo. 12 usa AM/PM. En España el formato 24h es el habitual.
displaySeconds false Oculta los segundos. El reloj muestra HH:MM sin los segundos. Si fuera true, el DOM se repintaría cada segundo. Sin segundos, solo se repinta cada minuto, reduciendo el trabajo del renderizador.
showDate true Muestra la fecha debajo de la hora. Si fuera false, solo se vería la hora.
dateFormat "dddd, D MMMM" Formato Moment.js para la fecha. Resultado: "lunes, 17 marzo". El formato es legible y no incluye el año (innecesario en un espejo del día a día). "DD/MM/YYYY" daría "17/03/2026". Los códigos de Moment.js: dddd=día semana completo, D=día número, MMMM=mes completo, YYYY=año 4 dígitos.
showWeek false Oculta el número de semana del año (ej: "Semana 11"). Solo útil si se trabaja con calendarios por semanas. Se omite para mantener la interfaz limpia.
timezone "Europe/Madrid" Fuerza la zona horaria a Madrid (España peninsular). Importante si el servidor corre en una máquina configurada con UTC. Sin este parámetro, usa la zona horaria del sistema operativo. Al poner "Europe/Madrid" el módulo funciona correctamente aunque el SO esté en UTC (como muchos servidores Linux).

9.2 · weather — tiempo actual y previsión

Se usan dos instancias del módulo weather en la misma posición (top_right): una para el tiempo actual y otra para la previsión de 5 días. Ambas tienen el mismo apiKey y location, pero difieren en el weatherEndpoint y algunos parámetros de visualización. MagicMirror apila los módulos en la misma posición verticalmente.

ParámetroValorQué hace y por qué
weatherProvider "openweathermap" Indica qué proveedor de datos usar. El módulo nativo soporta varios (darksky, weatherbit, etc). Se eligió OpenWeatherMap porque tiene plan gratuito sin tarjeta.
weatherEndpoint "/weather" / "/forecast" Ruta del endpoint de OpenWeatherMap. /weather devuelve el tiempo actual en ese instante. /forecast devuelve predicciones cada 3 horas para los próximos 5 días.
apiVersion "2.5" Versión de la API de OpenWeatherMap. La versión 2.5 es gratuita sin tarjeta de crédito. La 3.0 (One Call API) exige tarjeta aunque no se llegue al límite gratuito. Esta es la decisión técnica más importante de este módulo.
apiKey "de8f1dc0..." Clave de autenticación para OpenWeatherMap. Se obtiene registrándose en openweathermap.org. Se incluye en cada petición HTTP como parámetro &appid=. Sin ella, la API devuelve error 401 Unauthorized.
location "Valencia,ES" Ciudad y código de país ISO. El módulo la pasa como ?q=Valencia,ES a la API. Se incluye el código de país para evitar ambigüedad (hay una Valencia en Venezuela).
units "metric" Sistema de unidades. metric → temperatura en °C, viento en m/s. imperial → °F y mph. standard → Kelvin. Para España, siempre metric.
lang "es" Idioma de las descripciones meteorológicas. OpenWeatherMap traduce el texto ("cielo despejado", "lluvia ligera") al idioma solicitado. Sin este parámetro, daría en inglés.
updateInterval 600000 / 1800000 Milisegundos entre actualizaciones. El tiempo actual se refresca cada 10 minutos (600.000 ms). La previsión cada 30 minutos (1.800.000 ms). La previsión cambia más despacio, así que con menos peticiones es suficiente y se ahorra en la cuota de la API.
showWindDirection true Muestra la dirección del viento (N, NE, E...). Útil para saber si el viento viene del mar o de tierra.
showHumidity true Muestra el porcentaje de humedad relativa. Relevante en Valencia por el clima mediterráneo.
showFeelsLike true Muestra la temperatura sensación térmica (wind chill + humedad). Más útil que la temperatura real para decidir cómo vestirse.
type "forecast" Solo en la segunda instancia. Le indica al módulo que debe renderizar el widget de previsión (lista de días) en lugar del de tiempo actual.
maxNumberOfDays 5 Número de días a mostrar en la previsión. La API data/2.5/forecast devuelve hasta 5 días. Se muestran todos.
maxEntries 10 Máximo de entradas horarias a mostrar. La previsión tiene una entrada cada 3 horas; 10 entradas = ~30 horas de previsión detallada.
fade true Aplica un gradiente de opacidad a las entradas más lejanas en el tiempo. Las predicciones de dentro de 5 días son menos fiables; el fade visual lo comunica intuitivamente.

alert — No tiene parámetros de configuración propios. Es un módulo receptor: permanece inactivo hasta que otro módulo llama a la función interna sendNotification("SHOW_ALERT", { type, title, message, timer }). Entonces muestra un overlay a pantalla completa con el mensaje. Se incluye porque es necesario para el sistema de notificaciones interno de MagicMirror (algunos módulos lo usan para avisar de errores).

10Preguntas que puede hacer el jurado — y sus respuestas

Esta sección recoge las preguntas más probables agrupadas por tema, con respuestas directas y técnicamente correctas.

Arquitectura general

¿Por qué Node.js y no Python o Java?
Respuesta MagicMirror² está construido en Node.js, así que no es una elección libre sino una restricción del framework. Dicho esto, Node.js es adecuado: es excelente para servidores web con muchas conexiones concurrentes (gracias a su modelo de I/O no bloqueante), y el ecosistema npm tiene módulos para todo lo que necesita el proyecto (HTTP, WebSockets, OAuth). Electron también usa Node.js internamente para renderizar la interfaz.
¿Qué es Electron exactamente?
Respuesta Electron es un framework que permite crear aplicaciones de escritorio usando tecnologías web (HTML, CSS, JavaScript). Internamente combina Chromium (el motor del navegador Chrome) con Node.js. La interfaz del espejo es básicamente una página web que se renderiza a pantalla completa sin barras ni botones de navegador. Así MagicMirror puede usar HTML/CSS para el diseño y Node.js para la lógica de servidor, todo en un mismo proceso.
¿Qué diferencia hay entre el servidor de MagicMirror (puerto 8080) y VS Code Live Server (puerto 5500)?
Respuesta Son dos servidores web completamente independientes. El de MagicMirror es un servidor Express personalizado que sirve la aplicación, procesa notificaciones y expone la API REST. VS Code Live Server es una extensión del editor que sirve archivos estáticos del disco para previsualización en desarrollo — tiene recarga automática al guardar. En producción (en la Raspberry Pi o el PC del espejo) solo existe el servidor de MagicMirror; VS Code Live Server es solo una herramienta de desarrollo local.
¿Podría el espejo funcionar sin conexión a internet?
Respuesta Parcialmente. Los módulos que no dependen de APIs externas funcionarían sin internet: clock (usa el reloj del sistema), compliments (textos locales), MMM-pages, MMM-Remote-Control, MMM-Page-Indicator, alert y MMM-SmartWebDisplay (si las imágenes son locales, que es nuestro caso). Los que fallarían sin internet son weather (OpenWeatherMap), newsfeed (RSS), MMM-Jast (Yahoo Finance) y MMM-Spotify (API de Spotify). Estos simplemente dejarían de actualizar y mostrarían los últimos datos cacheados o una pantalla vacía.
¿Qué pasa si MagicMirror falla o se reinicia el sistema?
Respuesta Para eso está PM2. PM2 es un gestor de procesos de Node.js que monitoriza la aplicación. Si el proceso de MagicMirror muere por cualquier razón (error no capturado, crash), PM2 lo detecta y lo reinicia automáticamente. Con pm2 startup y pm2 save, PM2 se registra como servicio del sistema operativo y arranca con él. Así, cuando se enciende el PC del espejo, MagicMirror arranca solo sin intervención manual.

Sobre módulos y config.js

¿Cómo se añadiría un módulo nuevo al espejo?
Respuesta El proceso estándar es: 1) clonar o descargar el módulo en la carpeta modules/, 2) ejecutar npm install dentro de la carpeta del módulo para instalar sus dependencias, 3) añadir su entrada en el array modules[] de config.js con su nombre exacto (igual que la carpeta), posición y opciones de configuración, 4) reiniciar MagicMirror con pm2 restart MagicMirror. Ese es todo el proceso gracias a la arquitectura modular.
¿Por qué hay dos instancias del módulo weather en config.js?
Respuesta MagicMirror permite instanciar el mismo módulo varias veces con configuraciones diferentes. Se aprovecha esto para mostrar dos widgets distintos en la misma posición (top_right): uno con el tiempo actual (endpoint /weather) y otro con la previsión de 5 días (endpoint /forecast). Son dos instancias del mismo código base pero con parámetros distintos. MagicMirror los apila verticalmente en la misma columna.
¿Por qué los nombres de módulos son sensibles a mayúsculas?
Respuesta MagicMirror carga los módulos buscando la carpeta cuyo nombre coincida exactamente con el campo module: en config.js. En Linux y macOS, el sistema de archivos distingue entre mayúsculas y minúsculas: mmm-pages y MMM-pages son carpetas diferentes. Si el nombre en config.js no coincide exactamente, MagicMirror no encuentra la carpeta y el módulo no carga. En Windows no falla porque NTFS no distingue mayúsculas, pero es importante para la portabilidad a Raspberry Pi (Linux).
¿Qué es un módulo "fijo" en MMM-pages?
Respuesta Un módulo fijo (declarado en el array fixed de MMM-pages) es un módulo que permanece visible en todas las páginas. MMM-pages nunca le aplica display: none. Están ahí porque su información o funcionalidad es relevante siempre: newsfeed mantiene el contexto informativo continuo, MMM-Page-Indicator siempre muestra en qué página estás, MMM-Remote-Control necesita estar activo para recibir comandos de la API, y alert necesita estar disponible en caso de que cualquier módulo emita una notificación de alerta.

Sobre las APIs externas

¿Por qué OpenWeatherMap y no otra API del tiempo?
Respuesta El módulo nativo weather de MagicMirror soporta OpenWeatherMap de forma nativa con la versión 2.5 de la API, que es completamente gratuita sin necesidad de tarjeta de crédito. Otras APIs del tiempo o bien requieren pago o bien requieren adaptadores de módulo personalizados. El criterio fue: funcionalidad completa, coste cero, sin registro de métodos de pago. OpenWeatherMap cumple los tres.
¿Qué pasaría si se agota el límite gratuito de OpenWeatherMap?
Respuesta El plan gratuito de OpenWeatherMap (API 2.5) permite 60 peticiones por minuto y 1.000.000 de peticiones al mes. Con updateInterval: 600000 (cada 10 minutos) para el tiempo actual y 30 minutos para la previsión, el espejo hace en total unas 6 peticiones por hora, 144 por día, unos 4.320 al mes. Estamos muy lejos del límite. Si se llegara, la API devolvería error 429 Too Many Requests y el módulo dejaría de actualizar, pero no crashearía.
¿Por qué Yahoo Finance y no una API de cotizaciones oficial?
Respuesta Las APIs financieras oficiales (Alpha Vantage, Polygon, IEX Cloud) requieren registro y tienen límites bastante bajos en sus planes gratuitos, especialmente para datos en tiempo real. Yahoo Finance ofrece datos sin autenticación, sin límite conocido para consultas normales y con soporte para criptomonedas, acciones, ETFs y futuros. La contrapartida es que no hay garantía de estabilidad: Yahoo podría cambiar o cerrar esta API en cualquier momento. Es un compromiso consciente: gratuito y funcional ahora, potencialmente roto en el futuro.
¿Qué es RSS y por qué sigue usándose?
Respuesta RSS (Really Simple Syndication) es un estándar basado en XML que lleva activo desde 1999. Los medios de comunicación publican sus titulares en un archivo XML con una URL fija que cualquiera puede descargar. No requiere autenticación, no tiene límite de peticiones y no cambia: el formato es estable desde hace décadas. Los medios lo mantienen porque es la forma más sencilla de distribuir contenido a agregadores, apps y otros sistemas. Para este proyecto es ideal: gratuito, sin API key, múltiples fuentes, formato estándar.

Seguridad y comunicaciones

¿Por qué la API REST no tiene autenticación? ¿No es un riesgo?
Respuesta Se configuró secureEndpoints: false porque el espejo opera en una red local privada (192.168.18.x). En una red local doméstica, solo los dispositivos conectados al mismo router pueden acceder a esa IP. No está expuesto a internet. El riesgo es bajo y el beneficio es alto: cualquier dispositivo de la red (móvil, tablet) puede controlar el espejo sin tener que introducir credenciales. Si se quisiera exponer a internet (por ejemplo con un puerto forward en el router), sería imprescindible activar la autenticación con token configurando apiKey en MMM-Remote-Control.
¿Qué es OAuth 2.0 y por qué es necesario para Spotify?
Respuesta OAuth 2.0 es un protocolo de autorización que permite que una aplicación acceda a recursos de un servicio (en este caso Spotify) en nombre de un usuario, sin que ese usuario entregue su contraseña a la aplicación. El flujo es: el usuario autoriza la aplicación en la web de Spotify → Spotify entrega un código → la aplicación intercambia ese código por un token de acceso. El token tiene permisos limitados (solo los definidos en el scope) y caduca cada hora. Spotify lo exige porque si la aplicación tuviera la contraseña del usuario y sufriera una brecha, comprometería la cuenta entera. Con OAuth, lo máximo que se puede comprometer es el token, que caduca.
¿Qué diferencia hay entre access_token y refresh_token?
Respuesta El access_token es el token que se incluye en cada petición a la API de Spotify para demostrar que tienes permiso. Es de corta duración: caduca en 1 hora. El refresh_token es un token permanente (no caduca normalmente) que se usa exclusivamente para obtener un nuevo access_token cuando el anterior caduca. El usuario solo tiene que autorizar la app una vez; después, el módulo renueva el acceso automáticamente usando el refresh_token, sin que el usuario vuelva a intervenir.
¿Qué es X-Frame-Options y por qué bloqueaba las imágenes?
Respuesta X-Frame-Options es una cabecera HTTP de seguridad que un servidor puede enviar para indicar al navegador que no permita que su contenido sea mostrado dentro de un elemento <iframe>. Existe para prevenir ataques de clickjacking: técnica donde un atacante pone un sitio legítimo en un iframe invisible encima de su propio contenido, engañando al usuario para que haga clic en cosas sin saberlo. Muchos CDNs de imágenes envían esta cabecera. Nuestro workaround: en lugar de poner la URL de la imagen directamente en el iframe, se carga img.html (servido por localhost, sin esta restricción), y ese HTML usa una etiqueta <img> para cargar la imagen. Una etiqueta <img> no tiene restricciones de X-Frame-Options.
¿Qué es Socket.io? ¿No es lo mismo que WebSocket?
Respuesta WebSocket es el protocolo estándar del navegador para conexiones persistentes bidireccionales. Socket.io es una librería que usa WebSocket por defecto, pero añade funcionalidades extra: reconexión automática, fallback a HTTP long-polling si WebSocket no está disponible, y sobre todo el sistema de "rooms" y "eventos con nombre" que facilita el desarrollo. En MagicMirror, Socket.io permite que cada módulo emita notificaciones con nombre (como "WEATHER_DATA") y que solo los suscriptores a ese nombre las reciban. Con WebSocket puro habría que gestionar manualmente el enrutamiento de mensajes.

Técnicas y decisiones avanzadas

¿Cómo funciona exactamente el cambio de página sin recargar la web?
Respuesta MMM-pages no navega a ninguna URL nueva ni recarga el DOM. Todos los módulos están siempre presentes en el HTML de la página, cargados en memoria. Cuando llega la notificación PAGE_CHANGED con el índice de la nueva página, MMM-pages recorre su lista de módulos y aplica element.style.display = "none" a los que no pertenecen a esa página, y element.style.display = "block" a los que sí pertenecen. Es puro CSS con JavaScript. La transición de 1 segundo es un opacity en CSS. El resultado es un cambio inmediato y suave sin ningún coste de recarga de red.
¿Qué es un node_helper y por qué no puede hacer peticiones HTTP el módulo frontend directamente?
Respuesta Técnicamente, el módulo frontend (que corre en Chromium) sí podría hacer peticiones HTTP. Pero hay tres razones para separarlo: 1) Seguridad: la API key (de OpenWeatherMap, por ejemplo) no debería estar expuesta en el código del frontend, que el usuario puede ver con DevTools. En el node_helper corre en Node.js, fuera del contexto del navegador. 2) Módulos de Node.js: el node_helper puede usar módulos npm como axios, moment, fs, que no están disponibles en el entorno del navegador. 3) Arquitectura: la separación de responsabilidades hace el código más mantenible.
¿Por qué se usa Moment.js para las fechas y no el objeto Date de JavaScript?
Respuesta El objeto Date de JavaScript tiene una API poco ergonómica para formatear fechas y trabajar con zonas horarias. Moment.js permite escribir moment().format("dddd, D MMMM") en lugar de concatenar manualmente date.getDay(), date.getMonth(), etc. También maneja mejor la internacionalización y las zonas horarias con la extensión moment-timezone. MagicMirror usa Moment.js en varios módulos internos, por eso está disponible como dependencia global.
¿Por qué las imágenes de la galería están en una subcarpeta (imagenes_a_mostrar/) y no directamente en la carpeta del módulo?
Respuesta Separar las imágenes del código del módulo es una práctica de organización. La carpeta del módulo contiene archivos de código (.js, .css, package.json). Mezclar imágenes con código dificulta: 1) encontrar rápidamente qué es código y qué son recursos, 2) hacer actualizaciones del módulo sin borrar accidentalmente las imágenes, 3) gestionar el control de versiones (las imágenes suelen estar en .gitignore si son grandes). Con la subcarpeta, la separación es explícita.
¿Qué diferencia hay entre updateInterval y reloadInterval en el módulo newsfeed?
Respuesta Son dos ciclos de tiempo completamente distintos. updateInterval (7.500 ms) controla con qué frecuencia cambia el titular visible en pantalla: cada 7,5 segundos se pasa al siguiente titular de la cola local. reloadInterval (300.000 ms = 5 minutos) controla con qué frecuencia el módulo hace una petición de red para descargar el XML RSS de los medios y refrescar la cola con titulares nuevos. El primero es un cambio visual local; el segundo es una petición de red.
¿Cómo se añadiría una imagen nueva a la galería?
Respuesta El proceso es: 1) Copiar el archivo de imagen a modules/MMM-SmartWebDisplay/imagenes_a_mostrar/. 2) Añadir una nueva URL al array url en config.js siguiendo el mismo patrón: "http://localhost:8080/modules/MMM-SmartWebDisplay/img.html?src=http://localhost:8080/modules/MMM-SmartWebDisplay/imagenes_a_mostrar/nombre_nuevo.jpg". 3) Reiniciar MagicMirror con pm2 restart MagicMirror. No hay que tocar ningún otro archivo.
¿Por qué el espejo necesita que la pantalla sea retroiluminada y no solo cualquier pantalla?
Respuesta El efecto del espejo inteligente se consigue poniendo una pantalla retroiluminada detrás de un espejo semitransparente (vidrio con una capa fina de plata o espejo acrílico). La clave es el contraste: el fondo negro de MagicMirror no emite luz → el espejo lo refleja normalmente. Los elementos blancos o luminosos (texto, iconos) sí emiten luz → esa luz atraviesa el espejo semitransparente y es visible. En una pantalla normal encendida sin retroiluminación controlada (como una pantalla de papel electrónico), el efecto no funciona porque no hay suficiente diferencia de luminosidad.
¿Qué es MIT License y qué implica para este proyecto?
Respuesta La licencia MIT es una licencia de software libre extremadamente permisiva. Permite usar, copiar, modificar, distribuir e incluso vender el software, con la única condición de mantener el aviso de copyright original. Para este proyecto implica que se puede usar MagicMirror² y todos sus módulos sin coste, modificarlos si es necesario, y distribuir el resultado. No hay obligación de publicar las modificaciones (a diferencia de GPL). Es la licencia más común en el ecosistema Node.js/npm.
¿Cómo se escala este proyecto si se quisiera añadir más funcionalidades?
Respuesta La arquitectura modular de MagicMirror facilita la escalabilidad. Para añadir funcionalidades: 1) Si existe un módulo de la comunidad (hay más de 1.500), se instala y configura. 2) Si no existe, se crea un módulo nuevo con su archivo NombreModulo.js (frontend) y opcionalmente su node_helper.js (backend). 3) Para añadir más páginas, se añade un sub-array más en la configuración de MMM-pages y se actualiza pages: N en MMM-Page-Indicator. El núcleo de MagicMirror no se toca nunca; toda la funcionalidad vive en módulos independientes.

Conclusión

Este proyecto integra múltiples APIs externas, un sistema de autenticación OAuth 2.0, una API REST propia, una arquitectura cliente-servidor en tiempo real y una solución creativa a un problema de seguridad web (X-Frame-Options), todo sobre una plataforma modular de código abierto.

Cada decisión técnica tiene un motivo concreto: funcionalidad, compatibilidad, coste cero y facilidad de mantenimiento.
