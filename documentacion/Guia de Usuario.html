MagicMirror² — Guía de Usuario

Todo lo que necesitas para usar, controlar y entender tu espejo inteligente. Puerto local: 3000. Asegúrate de haber ejecutado arrancar.bat antes de abrir cualquier enlace.

1Arrancar y apagar

En la raíz de la carpeta tienes dos archivos .bat. Doble clic para usarlos.

▶
arrancar.bat
Inicia el servidor MagicMirror en el puerto 3000 y abre automáticamente el navegador en http://localhost:3000.
BAT
■
apagar.bat
Detiene el proceso del servidor liberando el puerto 3000. Úsalo siempre que quieras cerrar el espejo limpiamente.
BAT
Si ves la página en blanco o los módulos no cargan, espera 5-10 segundos y recarga. Node necesita un momento para inicializar todos los helpers.

4Cambiar de página

El espejo tiene 3 páginas gestionadas por el módulo MMM-pages. Puedes cambiar de página de tres formas:

1
Inicio
Reloj digital · Frases motivacionales · Tiempo actual Valencia · Previsión 5 días
2
Finanzas & Música
Bitcoin · Dogecoin · Petróleo Crudo (tiempo real) · Reproductor Spotify
3
Galería
Slideshow de imágenes locales (img1–img7) con rotación cada minuto

Método 1 — Panel visual

Abre el Panel de Páginas y haz clic en la página deseada.

Método 2 — API REST (fetch / curl)

// Ir a Página 1 (índice 0)
fetch('http://localhost:3000/api/notification/PAGE_CHANGED', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ payload: 0 })  // 0 = Inicio, 1 = Finanzas, 2 = Galería
})

// Página siguiente
fetch('http://localhost:3000/api/notification/PAGE_INCREMENT', { method: 'POST' })

// Página anterior
fetch('http://localhost:3000/api/notification/PAGE_DECREMENT', { method: 'POST' })

Método 3 — curl

curl -X POST http://localhost:3000/api/notification/PAGE_CHANGED \
     -H "Content-Type: application/json" \
     -d "{\"payload\": 1}"

iArranque del servidor — qué ocurre por dentro

Cuando ejecutas node serveronly (o arrancar.bat), se ejecuta esta cadena:

serveronly/index.js
    └── js/app.js                    ← núcleo de la aplicación
            ├── Lee config/config.js
            ├── Valida sintaxis
            ├── js/server.js          ← crea Express + Socket.IO
            │       ├── ipAccessControl()     ← whitelist de IPs ([] = todas)
            │       ├── express.static()      ← sirve /modules, /css, /js...
            │       └── server.listen(3000)
            └── Carga node_helpers de cada módulo
                    ├── MMM-Remote-Control/node_helper.js
                    ├── MMM-Jast/node_helper.js
                    ├── MMM-Spotify/node_helper.js
                    └── newsfeed/node_helper.js

El servidor es Express (HTTP clásico) con Socket.IO montado encima. Express sirve los archivos estáticos (HTML, CSS, JS de los módulos). Socket.IO es el canal en tiempo real entre Node y el navegador — sin él, los módulos no pueden comunicarse.

iCarga en el navegador — flujo cliente

Cuando abres http://localhost:3000 el navegador hace lo siguiente:

http://localhost:3000
    └── index.html
            └── js/main.js
                    ├── Conecta por WebSocket a Socket.IO
                    ├── GET /startup  → recibe la config serializada
                    ├── Carga cada módulo: su .js + su .css
                    └── Cuando todo está listo:
                            emite → DOM_OBJECTS_CREATED
                                └── MMM-pages recibe la notificación
                                        → oculta todos los módulos que NO son de página 0
                                        → muestra los de página 0

Los módulos no se cargan/descargan al cambiar de página. Todos existen siempre en el DOM. MagicMirror simplemente los muestra u oculta con animación CSS. Por eso el cambio de página es instantáneo.

iSistema de notificaciones — el sistema nervioso

Es el mecanismo que conecta todos los módulos entre sí. Hay dos tipos completamente distintos:

Tipo A — Notificaciones internas (módulo → módulo)

Solo viajan dentro del navegador, por JS puro. No pasan por el servidor.

// Un módulo envía:
this.sendNotification('PAGE_CHANGED', 1)

// Todos los módulos del espejo la reciben en:
notificationReceived(notification, payload) {
    if (notification === 'PAGE_CHANGED') { ... }
}

Así es como MMM-pages recibe la orden de cambiar de página, o como MMM-Page-Indicator sabe qué punto iluminar.

Tipo B — Socket notifications (módulo browser ↔ node_helper)

Viajan por WebSocket entre el navegador y el servidor Node. Necesarias cuando un módulo requiere datos externos.

// Módulo en el browser → su node_helper:
this.sendSocketNotification('FETCH_NEWS', { url: 'https://...' })

// node_helper hace el fetch real desde Node y responde:
this.sendSocketNotification('NEWS_ITEMS', { items: [...] })

// El módulo en el browser recibe:
socketNotificationReceived(notification, payload) { ... }
Los módulos del browser no pueden hacer peticiones a servicios externos directamente (restricciones CORS, sin acceso al filesystem). Por eso necesitan el node_helper como intermediario.

iCómo cambian las páginas — flujo completo

Este es el recorrido completo desde que pulsas un botón en PAGINAS.html hasta que el espejo cambia visualmente:

① Usuario pulsa "Finanzas" en PAGINAS.html
    │
    ▼
② fetch POST /api/notification/PAGE_CHANGED  { payload: 1 }
    │   (petición HTTP desde el navegador al servidor)
    ▼
③ MMM-Remote-Control / node_helper.js
    │   handleNotification() recibe la petición
    │   → this.sendSocketNotification('NOTIFICATION', { notification: 'PAGE_CHANGED', payload: 1 })
    ▼
④ WebSocket → llega al navegador del espejo
    │
    ▼
⑤ MMM-Remote-Control.js (en el browser del espejo)
    │   → this.sendNotification('PAGE_CHANGED', 1)   ← notificación interna
    ▼
⑥ MMM-pages.js  recibe en notificationReceived()
    │   this.curPage = 1
    │   this.updatePages()
    │       └── animatePageChange()
    │               ├── MM.getModules()
    │               │     .exceptWithClass(['MMM-Jast','MMM-Spotify'])
    │               │     .enumerate(m => m.hide(500ms))   ← oculta lo que NO es de pág 1
    │               └── setTimeout(500ms):
    │                     MM.getModules()
    │                       .withClass(['MMM-Jast','MMM-Spotify'])
    │                       .enumerate(m => m.show(500ms)) ← muestra lo de pág 1
    ▼
⑦ MMM-pages envía notificación interna NEW_PAGE → 1
    │
    ▼
⑧ MMM-Page-Indicator recibe NEW_PAGE → actualiza el punto activo en pantalla

Los nombres en withClass / exceptWithClass son literalmente los valores del campo module: en config.js. Por eso es crítico que coincidan exactamente con lo declarado en el array modules de MMM-pages.

iProbar conexión — cómo funciona el botón

El botón "Probar conexión" de PAGINAS.html ejecuta esto:

fetch('http://localhost:3000/api/module', { method: 'GET' })
ResultadoSignifica
HTTP 200 — "Conexión OK"El servidor está corriendo y la API responde correctamente
"Sin conexión: Failed to fetch"El servidor no está arrancado o el puerto 3000 no es el correcto
HTTP 4xx / 5xxEl servidor responde pero algo falla internamente — revisa los logs en la terminal

Se usa GET /api/module porque siempre devuelve datos sin efectos secundarios — es la prueba más segura y representativa de que todo el stack funciona.

iCómo funciona la API REST por dentro

MMM-Remote-Control/node_helper.js se engancha directamente al servidor Express principal mediante this.expressApp.get/post(...). No hay un servidor separado — todas las rutas /api/... viven en el mismo proceso Node que sirve el espejo.

// Dentro de node_helper.js:
this.expressApp.post('/api/notification/:name', (req, res) => {
    const notification = req.params.name   // ej: PAGE_CHANGED
    const payload      = req.body.payload  // ej: 1

    // Lo envía por WebSocket al browser del espejo:
    this.sendSocketNotification('NOTIFICATION', { notification, payload })

    res.json({ success: true })
})

El flujo completo de cualquier llamada API es siempre: HTTP → node_helper → WebSocket → módulo browser → notificación interna → acción visual.

iNode helpers — para qué sirven

Cada módulo que necesita datos externos tiene un archivo node_helper.js que corre en Node.js, separado del código del browser.

MóduloQué hace su node_helper
newsfeedDescarga y parsea los RSS feeds cada 5 min. El browser solo recibe los titulares ya procesados.
MMM-JastConsulta Yahoo Finance cada 30 s. Evita problemas CORS que tendría si lo hiciera el browser.
MMM-SpotifyGestiona el token OAuth (refresco automático) y consulta la API de Spotify.
MMM-Remote-ControlExpone todos los endpoints REST, gestiona el config.js y controla los módulos.
calendarDescarga y parsea ficheros .ics de calendarios remotos.

Los módulos sin node_helper (clock, compliments, MMM-pages, MMM-Page-Indicator) funcionan completamente en el browser y no necesitan datos externos.

iLock strings — por qué los módulos no se pisan

Cuando MMM-pages oculta un módulo, le pone un "candado" con su propio identificador. Esto evita que Remote Control u otro sistema lo muestre accidentalmente mientras el sistema de páginas lo tiene oculto.

// MMM-pages oculta con su lock string:
module.hide(500, callback, { lockString: 'MMM-pages' })

// Solo puede volver a mostrarlo quien tenga ese mismo lock string:
module.show(500, callback, { lockString: 'MMM-pages' })

Sin lock strings, si usas Remote Control para ocultar un módulo y luego cambias de página, MMM-pages lo mostraría igualmente al intentar mostrar los módulos de esa página. Los lock strings garantizan que cada sistema controla solo lo que él mismo ocultó.
